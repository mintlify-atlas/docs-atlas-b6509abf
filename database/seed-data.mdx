## Role Catalog (Pre-Seeded)

The `roles` table is automatically populated when running `02_roles.sql`:

```sql
INSERT INTO roles (nombre_rol, descripcion) VALUES
    ('Estudiante',   'Usuario matriculado en un programa académico'),
    ('Empleado',     'Trabajador vinculado directamente a la UCC'),
    ('Contratista',  'Proveedor externo con contrato activo');
```

**Verify:**
```sql
SELECT * FROM roles ORDER BY id;
```

**Expected result:**
```
id | nombre_rol  | descripcion
---|-------------|------------------------------------------
1  | Estudiante  | Usuario matriculado en un programa académico
2  | Empleado    | Trabajador vinculado directamente a la UCC
3  | Contratista | Proveedor externo con contrato activo
```

<Note>
  These three roles are fixed and should not be modified during normal operations.
</Note>

## Verify Test Data

### Check Users and Roles

```sql
SELECT
    u.id_institucional,
    u.nombre_completo,
    u.acceso,
    u.total_fallas,
    STRING_AGG(r.nombre_rol, ', ' ORDER BY r.nombre_rol) AS roles
FROM usuarios u
LEFT JOIN usuario_roles ur ON ur.id_institucional = u.id_institucional
LEFT JOIN roles r ON r.id = ur.rol_id
GROUP BY u.id_institucional, u.nombre_completo, u.acceso, u.total_fallas
ORDER BY u.id_institucional;
```

## Generate Large Test Datasets

### Python Script for Bulk Data

```python
import csv
import random
from faker import Faker

fake = Faker('es_CO')  # Colombian Spanish locale

# Generate 1000 test users
with open('usuarios_bulk.csv', 'w', newline='', encoding='utf-8') as f:
    writer = csv.writer(f)
    writer.writerow(['id_institucional', 'documento_identidad', 'nombre_completo'])
    
    for i in range(1, 1001):
        id_inst = f'80300{i:04d}'
        doc_id = f'{random.randint(1000000000, 1099999999)}'
        name = fake.name()
        writer.writerow([id_inst, doc_id, name])

print("Generated usuarios_bulk.csv with 1000 records")

# Generate 600 students
with open('estudiantes_bulk.csv', 'w', newline='', encoding='utf-8') as f:
    writer = csv.writer(f)
    writer.writerow(['id_institucional', 'programa'])
    
    programas = [
        'Ingeniería de Sistemas', 'Derecho', 'Medicina', 'Administración',
        'Contaduría', 'Psicología', 'Arquitectura', 'Economía'
    ]
    
    for i in range(1, 601):
        id_inst = f'80300{i:04d}'
        programa = random.choice(programas)
        writer.writerow([id_inst, programa])

print("Generated estudiantes_bulk.csv with 600 records")

# Generate 300 employees
with open('empleados_bulk.csv', 'w', newline='', encoding='utf-8') as f:
    writer = csv.writer(f)
    writer.writerow(['id_institucional', 'cargo', 'dependencia'])
    
    cargos = ['Docente', 'Coordinador', 'Secretaria', 'Auxiliar', 'Decano']
    dependencias = [
        'Facultad de Ingeniería', 'Facultad de Derecho', 'Registro Académico',
        'Biblioteca', 'Vicerrectoría', 'Recursos Humanos'
    ]
    
    for i in range(601, 901):
        id_inst = f'80300{i:04d}'
        cargo = random.choice(cargos)
        dependencia = random.choice(dependencias)
        writer.writerow([id_inst, cargo, dependencia])

print("Generated empleados_bulk.csv with 300 records")
```

## Testing Automatic Blocking

### Test Case: Block User at 4 Failures

```sql
-- Create test user
INSERT INTO usuarios (id_institucional, documento_identidad, nombre_completo)
VALUES ('TEST001', '9999999999', 'Usuario Prueba Bloqueo');

-- Initial state: activo, 0 failures
SELECT acceso, total_fallas FROM usuarios WHERE id_institucional = 'TEST001';
-- Expected: activo, 0

-- Add 3 failures (should remain active)
INSERT INTO fallas (id_institucional, motivo) VALUES
    ('TEST001', 'olvido'),
    ('TEST001', 'olvido'),
    ('TEST001', 'olvido');

SELECT acceso, total_fallas FROM usuarios WHERE id_institucional = 'TEST001';
-- Expected: activo, 3

-- Add 4th failure (should auto-block)
INSERT INTO fallas (id_institucional, motivo) VALUES ('TEST001', 'olvido');

SELECT acceso, total_fallas FROM usuarios WHERE id_institucional = 'TEST001';
-- Expected: bloqueado, 4
```

<Info>
  The trigger `fn_actualizar_fallas()` automatically blocks the user when `total_fallas >= 4`.
</Info>

## Cleanup Test Data

### Remove All Test Data

```sql
-- Remove test users (cascades to all related tables)
DELETE FROM usuarios WHERE id_institucional LIKE 'TEST%';
DELETE FROM usuarios WHERE id_institucional LIKE '80100%';  -- Sample data
DELETE FROM usuarios WHERE id_institucional LIKE '80200%';  -- CSV imports
DELETE FROM usuarios WHERE id_institucional LIKE '80300%';  -- Bulk data

-- Reset semester
DELETE FROM semestres;
```

<Warning>
  This will delete all test users and their associated data (roles, failures, etc.).
</Warning>

---

### Selective Cleanup

```sql
-- Remove only failures (keeps users)
DELETE FROM fallas WHERE id_institucional LIKE 'TEST%';

-- Remove only student info (revokes Estudiante role via trigger)
DELETE FROM info_estudiante WHERE id_institucional LIKE 'TEST%';

-- Remove only blocked users
DELETE FROM usuarios WHERE acceso = 'bloqueado';
```

## Seed Data Best Practices

<AccordionGroup>
  <Accordion title="Use Prefixes for Test Data">
    Use distinguishable ID prefixes:
    - `801xxxxx` - Manual test users
    - `802xxxxx` - CSV imports
    - `803xxxxx` - Bulk generated data
    - `TEST*` - Temporary test users
    
    Makes cleanup easier.
  </Accordion>
  
  <Accordion title="Include Edge Cases">
    Test data should include:
    - Users with 0 failures
    - Users near blocking threshold (3 failures)
    - Users already blocked (4+ failures)
    - Single-role users
    - Multi-role users
    - Users with no roles
  </Accordion>
  
  <Accordion title="Keep Seed Scripts in Version Control">
    ```bash
    git add database/supabase/seed_test_data.sql
    git commit -m "Add seed data for testing"
    ```
    
    Team members can reproduce the same test environment.
  </Accordion>
  
  <Accordion title="Separate Production Seed from Test Seed">
    - `seed_production.sql` - Minimal data for production (roles only)
    - `seed_test_data.sql` - Development and testing data
    - `seed_bulk_data.sql` - Large dataset for performance testing
  </Accordion>
</AccordionGroup>

## Production Data Loading

For production deployment:

<Steps>
  <Step title="Load Minimal Initial Data">
    Only load the role catalog (already done in `02_roles.sql`).
  </Step>
  
  <Step title="Prepare Real User Data">
    Export from institutional systems:
    - Student list from academic registration system
    - Employee list from HR system
    - Contractor list from procurement system
  </Step>
  
  <Step title="Validate CSV Format">
    Ensure CSVs match table schema:
    ```
    usuarios: id_institucional, documento_identidad, nombre_completo
    info_estudiante: id_institucional, programa
    info_empleado: id_institucional, cargo, dependencia
    info_contratista: id_institucional, empresa
    ```
  </Step>
  
  <Step title="Load in Correct Order">
    1. `usuarios.csv`
    2. `estudiantes.csv` (triggers assign roles)
    3. `empleados.csv` (triggers assign roles)
    4. `contratistas.csv` (triggers assign roles)
  </Step>
  
  <Step title="Verify Data Integrity">
    ```sql
    -- Check all users have at least one role
    SELECT u.id_institucional, u.nombre_completo
    FROM usuarios u
    LEFT JOIN usuario_roles ur ON ur.id_institucional = u.id_institucional
    WHERE ur.id IS NULL;
    
    -- Should return empty result
    ```
  </Step>
</Steps>
